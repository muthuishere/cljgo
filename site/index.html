Clojure, hosted on Go.

A compiler written in Go that AOT-emits plain Go source — the ClojureScript model, with Go as the JavaScript — plus a tree-walk evaluator that is the REPL and the macro engine. One shared analyzer feeds both, and the same source runs interpreted at the prompt and compiles to a static native binary with byte-identical output.

Why cljgo

Five priorities, in order — the design mandate, not a wishlist.

1

Universal interop

Any Go module is importable and callable with zero hand-written bindings — the Go ecosystem is the standard library. C reaches in via cgo modules and purego FFI.

2

REPL-driven development

The tree-walk evaluator is a real Clojure REPL: live re-def, defmacro at the prompt, namespaces, nREPL for CIDER/Calva.

3

Faithful Clojure

Persistent data structures with real structural sharing, macros as plain fns, seq abstraction, vars as the indirection layer.

4

Performance in both modes

Fast tree-walk and a compiled performance ladder — a feature, not an option. Benchmarked in CI; perf regressions are treated like conformance failures.

5

cgo is first-class

CGO_ENABLED=1 projects are supported, not tolerated — cgo-based Go modules (sqlite drivers, sensors, GUI/audio) work like any other import.

Dual-mode is the gate

Interpreted result == compiled result. A dual-harness conformance suite enforces it on every commit; a REPL↔binary divergence is a release blocker.

Quickstart

Build from source is the known-good path — it needs a Go 1.26 toolchain (and a C toolchain with CGO_ENABLED=1 for cgo interop features).

1 · Build the CLI from source

# requires Go 1.26+
git clone https://github.com/muthuishere/cljgo
cd cljgo
go build -o cljgo ./cmd/cljgo   # -> ./cljgo
go install github.com/muthuishere/cljgo/cmd/cljgo@latest installs the cljgo binary too. Note that cljgo build currently needs the cljgo source tree available for its generated go.mod replace directive (via -runtime or $CLJGO_SRC), so the clone-and-build path is the safe default.

2 · Start the REPL

$ ./cljgo repl
cljgo 0.0.1-m2
user=> (defn greet [who] (str "hello, " who))
user=> (greet "cljgo")
"hello, cljgo"

3 · Write a file, run it, then compile it

hello.clj

;; hello.clj
(defn fact [n]
  (if (< n 2)
    1
    (* n (fact (- n 1)))))

(println "hello from cljgo")
(println "(fact 10) =" (fact 10))

interpret, then build a native binary

$ ./cljgo run hello.clj
hello from cljgo
(fact 10) = 3628800

$ ./cljgo build hello.clj   # -> ./hello
$ ./hello
hello from cljgo
(fact 10) = 3628800          # byte-identical

CLI surface

cljgo repl                       # start a REPL
cljgo run <file.clj>             # evaluate a file (interpreted)
cljgo build [-o out] <file.clj>  # compile a file to a native binary
cljgo check <file.clj> [--json]  # analyze, report diagnostics (ADR 0015)
cljgo explain <code> [--json]    # show an error code's explain page
cljgo version                    # print the version string

Zero-binding Go interop

require-go pulls in any Go package and calls it directly — no wrappers, no generated stubs. The Go toolchain is the classpath. This snippet is from examples/interop/core.clj and runs identically interpreted and compiled. 

(require-go '[strings])
(require-go '[strconv])
(require-go '[math])

;; single-return package fns + int64->int arg coercion
(println "ToUpper:" (strings/ToUpper "hello"))
(println "Repeat:"  (strings/Repeat "ab" 3))

;; a const in value position
(println "Pi:" math/Pi)

;; (T, error) call -> [v err] vector; happy-path err slot is nil
(println "Atoi ok:" (strconv/Atoi "123"))

;; the ! suffix unwraps-or-throws; here it returns the widened int64
(println "Atoi! ok:" (strconv/Atoi! "456"))

Errors as values

(T, error) Go calls shape to a [v err] vector — branch on the error slot, Clojure-style. The ! suffix unwraps or throws.

Members & concurrency

(.Method r …), (.-Field r), (pkg/T. {…}) constructors, plus (chan) / (>! c v) / (<! c) / (go …) over real goroutines — no CPS rewrite.

One analyzer, two backends

The unforgivable failure mode is the REPL diverging from the compiled binary. cljgo makes that structurally hard: one reader, one analyzer, one AST — feeding both a tree-walk evaluator and a Go-source emitter.

Reader → Analyzer → AST shared
Full syntax-quote reader, macroexpansion, one analyzer producing a single AST. Everything below consumes this — ADR 0002.
↓
Tree-walk evaluator pkg/eval
IS the REPL and the macro engine. Pre-resolved locals, non-allocating fast paths. Live re-def, defmacro at the prompt.
Go-source emitter pkg/emit
AOT-emits plain Go from go/packages type facts — direct, non-reflective calls. Then the Go toolchain produces a static native binary.
↓
Dual-harness conformance
Every semantic test runs through both paths, oracle-cited against JVM Clojure. Interpreted result == compiled result, or the build fails — ADR 0007.

Roadmap — the Zig model

The strategic direction (design/08): cljgo's "batteries" are Zig's, not Leiningen's/deps.edn's.

  • build.cljgo — the build is a program, not a data file. (defn build [b] …) at the project root defines an artifact/step DAG; cljgo build / build run / test mirror zig build. Replaces deps.edn for the AOT product (ADR 0021).
  • comptime — Zig-style compile-time value execution alongside Clojure macros. Macros transform syntax; comptime computes values embedded as literals. In the REPL, compile-time == eval-time (ADR 0009).
  • clojure.core compliance — the jank clojure-test-suite as the external yardstick, with a coverage ratchet in CI. Measured baseline: 34/242 suite files passing (14.0%), 51.2% of tested clojure.core vars resolved (2026-07-15) — climbing per batch (ADR 0022).
  • Cross-compilation — cljgo emits plain Go, so pure-Go + purego programs build for any OS/arch with no target toolchain. --target os/arch or a matrix per artifact.
