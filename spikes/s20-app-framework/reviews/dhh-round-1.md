# DHH-persona review — round 1 (fresh reviewer, verbatim, unsoftened)

1. **The name is fine, which worries me.** "keel" — the part of the ship nobody sees, that never steers. You've named your framework after your own abdication. Rails is called Rails because it puts you ON rails — it has opinions about where you're going. A keel just sits there in the cold water being structurally load-bearing while the captain does all the actual work. You picked a metaphor for "we won't help you steer" and you're proud of it. At least "sangam" had a soul; you rejected it because it "reads as a product name." Yes! A framework IS a product! That's the whole thesis you claim to have studied!

2. **There is no `keel new`. Full stop, this is disqualifying.** I read four documents and not one of them tells me what a beginner TYPES. Rails' first fifteen minutes is `rails new blog && bin/rails server` and you're looking at a running page before your coffee cools. Keel's first fifteen minutes is: create a directory (what layout? unspecified), create `conf.edn` (what goes in it? unspecified), declare a config schema (WHERE? The golden path calls `config/load!` and the schema it validates against is invisible — it's load-bearing and it's not on the page), create the `:users` table (there's a `db/insert` into a table no migration on this page created), then hand-write `-main`. You wrote "this page is the product" over a page that cannot actually run. The product is vaporware with beautiful typography.

3. **The golden path is a wiring diagram, not an application.** Count what the beginner writes before their first route: config load, pool construction, cache construction, queue construction — four defs of infrastructure plumbing, each threaded by hand into every function that needs it. `(defn signup [req] (let? [user (db/insert pg ...)]))` — `pg` is a top-level global the handler reaches for. So your grand omakase position is... globals and manual parameter threading. Rails figured out in 2004 that `User.create` beats passing a connection handle through every function like a bucket brigade. You'll say "explicit is better." Explicit is what you say when you've made the developer do the framework's job and want a medal for it.

4. **"Library style: you call it, it never calls you" is a cop-out dressed as a doctrine.** It's the framework equivalent of a restaurant bragging that the chef never seasons your food. The ENTIRE value of Rails is that it calls you — at the right time, in the right order, with the right defaults — so ten thousand apps share one shape and any Rails developer can walk into any Rails codebase and know where everything lives. Keel apps will each wire `-main` differently, thread their handles differently, name their config keys differently. You haven't avoided the framework wars; you've guaranteed every app IS its own tiny framework war. "No conventions about structure" is not a feature. It's the disease you claimed you were curing — you even quote the diagnosis ("the missing 20% was curation and conventions") and then ship the missing 20% missing.

5. **A beginner must learn your Result/Option algebra before "hello world."** Line 31 of the demo: `let?`. So before I've served my first request I need to understand railway-oriented programming, the ok/err wrapping, the short-circuit semantics, AND your parallel `!` universe where every function has an evil twin that throws. That's a fork in the road on every single call — "do I want `db/insert` or `db/insert!`?" — handed to someone who doesn't yet know what a route is. Rails lets you write `User.create` and learn about `create!` when you're ready to care. You've made the monad tutorial a prerequisite for the landing-page demo. The spec even calls exceptions "ceremony" while making the beginner carry a two-track error model in their head. Physician, heal thyself.

6. **Jobs handlers are "sealed at start" in a framework whose headline feature is live redefinition.** You measured — MEASURED, with benchmarks, PASS stamps — that HTTP handlers behind vars can be re-`def`ed against a live server, it's your proudest demo. Then one def later, `jobs/start` takes a handler map and seals it. So the REPL story is: web handlers are alive, job handlers are dead. Why? No reason given. It's an inconsistency a beginner will hit in week one ("I re-defed my job fn and nothing changed") and it exists because two sections of the spec were written by two moods. Put vars in the map, deref at dispatch — you already paid the 1.7ns, you bragged about the receipt.

7. **A web framework that cannot produce a web page.** "No HTML templating (owner: not a focus)." I'm sorry — what does `http/created user` return to a human being? JSON, presumably, though the spec never says. You've built keel.ai — a whole PILLAR for calling LLM APIs, with cross-provider fallback chains and price-tier matching — before you've built the ability to render `<h1>Hello</h1>`. That is the priority ordering of someone building for the 2026 conference talk, not for the person making an actual product. Rails shipped ERB before it shipped anything else because the first thing a human wants is to SEE their app. Your beginner's fifteen minutes ends with curl. Nobody falls in love over curl.

8. **The config story fails its own "misconfigured deploy must not boot" slogan at authoring time.** Fail-loud validation is right — but the schema that makes it work appears nowhere in the golden path, no default `conf.edn` is generated (because nothing is generated, see point 2), and `APP_DB_HOST → [:db :host]` means the mapping between env var and config path is a convention the operator must reverse-engineer at 2 a.m. Where's `keel config` to print the resolved map with each key's winning layer? You designed the boot-time failure beautifully and left the debugging story to archaeology. Also, "four layers, not Spring's fourteen" — congratulating yourself for being less bad than Spring is the lowest bar in this industry, and you cleared it by lying down.

9. **No sessions, no cookies, no forms, no CSRF, no auth — auth is "an essay."** The spec's word is "reference design chapter." So the single most security-critical, most-universally-needed piece of every application ever built is homework. Every keel app will hand-roll JWT middleware from a blog post — YOUR blog post — with hand-rolled bugs. Rails ships `has_secure_password`, sessions, CSRF protection, signed cookies, because the framework taking responsibility for the sharp edges is what "batteries included" MEANS. "We wrote an essay explaining why there's no battery" is a Spring-committee move wearing a Rails t-shirt.

10. **The omakase claim collapses under inspection.** Omakase means the chef DECIDES. Real decisions the beginner still makes with keel: project layout (undecided), where the config schema lives (undecided), Result vs `!` on every call (their problem, forever), SQL strings vs "HoneySQL-style data, both first-class" — BOTH first-class is the opposite of a blessed way, it's a menu — how to structure anything above `-main` (undecided), how to test any of it (the word "test" appears in this spec only to describe the framework's own conformance suite — the APP developer's testing story is nonexistent). You blessed the plumbing and shrugged at everything the developer touches daily.

11. **Docs-as-product gets one sentence and a prayer.** Your own analysis says the reason Clojure never got its Rails is that "conventions, docs, curation — the expensive part nobody funded." Then the proposal funds: tiers T1–T4 of code, conformance tests, perf budgets — and zero deliverables for guides, generators, a tutorial, error-message quality, or the `--help` output. The Rails Guides are why Rails won; they are a product with their own maintainers. You've written the "why frameworks fail" essay and then filed the failure under Non-goals by omission.

12. **`(http/health {:db pg :jobs q})` is the only reason `q` exists on the page, and it's telling.** The golden path defines a job queue whose queue handle is used once, decoratively, in a health check. Meanwhile the actual hard operational questions — graceful drain on deploy, what happens to in-flight jobs on SIGTERM, connection-pool sizing, timeouts ANYWHERE (not one timeout appears in four documents describing an HTTP server, a DB pool, a job runner, and LLM calls) — are unaddressed. You demoed the happy path of the happy path. Production is where frameworks earn their name, and this spec's production story is "Postgres is blessed."

13. **1.6× interpreted overhead is fine; hiding the REPL gap behind it is not.** The ADR quietly admits the interpreter can't even see `net/http` yet — "S20's adapter had to be Go precisely because of this gap." So the live-coding demo that anchors the entire pitch currently runs on a bridge the beginner won't have. The centerpiece feature is scheduled, not shipped, and the VERDICT presents it in the past tense with PASS stamps. That's not a golden path, that's a rendering of one.

14. **"Survived three adversarial DHH-persona review rounds" — with the rounds recorded as "(recorded after the round)" and "Open questions: (populated after round 3)".** You cited your own adversarial review as evidence of hardening while the review sections are literally empty placeholders. This is the enterprise pattern I hate most of all: the process artifact that attests to rigor that hasn't occurred. Spring has a committee; you have a committee of imaginary me, and even imaginary me hasn't shown up to the meetings yet.

15. **What I'd actually demand.** (a) `keel new myapp` that emits a running app — layout, conf.edn, schema, first migration, first test — and `keel dev` that boots it with the REPL attached; nothing else in this spec matters until that exists, so it is T0, not absent. (b) A blessed project layout, in the spec, with the same normative SHALL language you lavished on singleflight caches. (c) `!` variants as the beginner default in every doc and demo, `let?` introduced in chapter 4 when they've earned a reason to care. (d) HTML out of the box — even one hiccup-style function — before one line of keel.ai. (e) Sessions + CSRF + password auth as code, not literature. (f) The Guides as a tracked deliverable with the same gates as the code.

**Verdict.** Would this make me quit Rails? No — and the tragedy is that it could have made me nervous. The substrate is genuinely enviable: static binaries, real goroutines, live vars at nanosecond cost, zero-binding access to the entire Go ecosystem, transactional jobs in Postgres with no broker — I'd trade Sidekiq's Redis babysitting for that seam tomorrow. But you've taken this once-a-decade substrate and specced a box of premium components with no chassis, no key in the ignition, and a note where the steering wheel should be explaining why steering is a personal journey. The spec optimizes for the reviewer who reads one page of code and nods, not for the nineteen-year-old at midnight who needs the framework to have already made a hundred decisions so she can make the three that matter. Rails won because it was arrogant on the user's behalf. Keel, true to its name, is content to sit under the waterline and hold things straight while the beginner drowns above deck. Ship `keel new`, ship the guides, ship a web page, seal nothing you claim is live — or accept that you've built another very good Clojure library collection, which is exactly the thing your own ADR says has failed for twenty years.
