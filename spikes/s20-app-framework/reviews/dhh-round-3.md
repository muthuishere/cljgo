# DHH-persona review — round 3 (fresh reviewer, verbatim, unsoftened)

1. **The first fifteen minutes require the user to install Postgres, and nobody wrote that sentence down.** The whole draft is built on `cljgo new && cljgo dev` being magic — and then T0 generates `migrations/0001_users.sql` and `cljgo dev` "applies migrations." Applies them to *what*? There is no dev-database story anywhere in five documents. No embedded database, no SQLite, no `cljgo dev --db`, no compose file in the scaffold, nothing. So the actual first fifteen minutes are: install Postgres, create a database, figure out the connection URL format, set `APP_DB_URL`, *then* run the magic command. Rails ships SQLite as the dev default for precisely this reason and has for twenty years. This isn't a gap, it's the crater where the beginner falls in at minute two. Demand: `cljgo dev` must produce a working, persistent database with zero installation — embedded Postgres, SQLite behind the same `keel.db` face, a managed container, I don't care which, but pick one and bless it, or the T0 tutorial is fiction.

2. **The golden page violates the spec's own crown-jewel requirement, on line five.** The spec thunders: "Framework constructors SHALL perform no I/O at construction… requiring `app.main` under `APP_PROFILE=test` attempts no network connection." Then the golden page does `(def q (jobs/start! pg {...}))` at the top level. `start!` — your own bang convention! — starts goroutine workers doing LISTEN/NOTIFY against Postgres. That is I/O at namespace load, in the sacred artifact, blessed as the conformance test. Either `jobs/start!` is secretly lazy (in which case the name is a lie and the "no-I/O" contract is enforced by vibes), or requiring `app.main` in a test dials your database. Pick one; right now the document set contradicts itself in the one place it swore it never would.

3. **"SIGTERM drains requests AND jobs" — through what wire, exactly?** `http/serve` receives `routes` and `{:port ...}`. It never receives `q`. Yet the golden page's comment promises serve's shutdown drains the job queue. The only place `q` appears near http is inside a health-check handler, which is just data in a route. So either there's an ambient global registry where `jobs/start!` secretly registers itself for shutdown — the exact "nothing ambient" sin this framework's entire identity is built on renouncing — or the comment is false advertising. A framework whose founding creed is "no hidden call graph" has a hidden call graph in its hello-world comments. Demand: shutdown composition must be visible on the page (`(http/serve routes {:port p :drain [q]})` or similar), or delete the claim.

4. **CSRF-by-default breaks the golden page's own POST, and no document notices.** The proudest scenario in the spec: "a state-mutating POST without a valid CSRF token is rejected — with zero middleware code in the app." Wonderful. Now look at the page: `POST /signup` takes a JSON body. Where does the beginner's curl command — the one the 15-minute tutorial must contain — get a CSRF token? `keel.html` has no form helper, no token emission, nothing that ever renders a token into markup, and the page renders zero forms anyway. So the tutorial's first POST returns 403 and the beginner's first experience of "security is what you didn't type" is being locked out of their own app by machinery they can't see. Rails solved this — form helpers embed the token; JSON APIs opt out explicitly. This draft blessed the lock and forgot to mint any keys. Demand: `html/form` that emits the token, plus a specified, explicit posture for JSON endpoints — before T1 ships, because right now the security scenario and the tutorial scenario cannot both pass.

5. **The spec contradicts itself on what happens when a handler returns a Result — in adjacent requirements.** Requirement "Errors have one beginner surface": "A handler that returns a raw Result value SHALL fail loudly in dev mode (500), never be silently coerced." Requirement "Data layer," scenario "railway signup (day two)": the `let?` chain "short-circuits with `(err e)` and the funnel renders it, no exception ceremony." Both handlers return `(err e)`. One gets a helpful 422; the other gets a scolding 500. There is no mechanism anywhere in five documents that distinguishes them — no wrapper on the day-two page, no `http/result` funnel form, nothing. This is not a nitpick; it's the framework's advertised growth path, and as written it's a coin flip. Demand: the day-two page must show the visible thing that makes the funnel legal (`(http/railway (let? ...))`, whatever), or the loud-500 rule dies. You cannot ship both sentences.

6. **Wholesale middleware replacement means the safe stack evaporates on day three.** "Passing `:middleware` replaces it wholesale, no merge magic." Cute slogan. Real life: on day three someone adds request-ID middleware, types `:middleware [request-id]`, and silently drops CSRF, sessions, recover, and the access log. The framework's marquee feature — "the security is what you didn't type" — is destroyed by the first thing every real app types. Rails treats the middleware stack as inspectable, spliceable data (`insert_before`) because we learned this lesson in public. Nothing in the spec even says `(http/defaults)` returns a vector you can `conj` onto. Demand: defaults are DATA, printed by `cljgo config`-style introspection, spliceable by name, and dev mode screams when a replacement stack lacks CSRF/recover. Otherwise "no merge magic" is just "footgun, artisanal."

7. **Two error dialects means every real codebase is a pidgin.** Beginners live in `!`; day two you "drop the `!`" and rewrite handlers into `let?`. So the upgrade path is a rewrite, per handler, forever — and every team ends up with signup in railway style and show-user in throw style, and every code review relitigates which surface this function deserves. Rails has one way to fail out of a controller. The one-rule slogan ("plain = Result, `!` = throws") is a naming convention wearing a philosophy costume; it doesn't tell anyone *when to switch*, and "when you earn a reason" is horoscope-grade guidance. Demand: one blessed surface for app code, with the other documented as the escape hatch — omakase means the chef picks the fish, not "here are two fish, feel your feelings."

8. **The tiers contradict the tasks: T0 applies migrations, but migrations ship in T2.** Task 0.2: `cljgo dev` "applies pending migrations." Task 2.3: `cljgo migrate` is built in tier TWO. So at T0 the generator writes `migrations/0001_users.sql` and the dev command applies it using machinery that doesn't exist for two more tiers — against a data layer (`keel.db`, also T2) that also doesn't exist. Either T0 quietly contains half of T2, or the T0 scaffold generates a dead file and a dev command that lies about what it did. The tier structure is the project plan; it doesn't currently topologically sort. Demand: redraw the tiers so every T0 verb has a T0 implementation, and say what the T0 `main.cljg` actually contains — because "the golden page trimmed of unshipped pillars" at T0 is config, two routes, and a stylesheet, which is a fine static page and a very small miracle.

9. **Dev runs `:memory` jobs, prod runs Postgres jobs — dev/prod parity sacrificed on page one of tier three.** Task 3.2: the `:memory` backend is "used by the scaffold in dev." The entire argument for the Oban model is that jobs are durable rows that survive restarts and enqueue transactionally with your writes. In dev, per the scaffold, none of that is true — `enqueue!` on a tx handle committing "atomically" with a memory channel is atomic with nothing. And the golden page hardcodes `(jobs/start! pg ...)`, so the swap mechanism isn't even shown. Which is it? Demand: dev uses the real Postgres backend (you already required Postgres in point 1 — spend it), and `:memory` is for tests only. Anything else means the first time transactional enqueue matters is in production.

10. **"No ambient globals" — the golden page is three ambient globals.** `cfg`, `pg`, `mem`, `q`: top-level defs that every handler closes over. Fine, honest, Clojure-flavored. But then the testing story says "per-test tx rollback fixtures" — against handlers that reach for a namespace-level `pg` baked in at load time. How does the fixture swap the pool? `with-redefs` on the golden page's vars? Rebinding? Passing a system map that the page conspicuously doesn't use? Five documents, zero answers. The framework's proudest structural claim (explicit handles) and its proudest testing claim (rollback fixtures) have never been introduced to each other. Demand: show the generated test. Byte for byte. If it needs `with-redefs #'app.main/pg`, put that ugliness on the table now.

11. **The ORM ban left the naming conventions to astrology.** The page reads `(:created-at user)` from a row produced by `select *` against a column that Postgres will call `created_at`. So somewhere, `keel.db` is renaming snake_case columns to kebab keywords — a silent, framework-level data transformation that no requirement specifies, in the framework whose motto is "SQL strings in, plain maps out, nothing hidden." Same for the env overlay: does `APP_DB_POOL_SIZE` become `[:db :pool :size]` or `[:db :pool-size]`? Underscores are doing double duty as both nesting and key-spelling, and the spec shrugs. These are exactly the ten thousand tiny conventions that ARE a framework — Rails is mostly an opinion about names. Demand: a written names doctrine (column↔keyword, env↔path, table keywords in `insert!`) with conformance tests, or the first debugging session is a scavenger hunt.

12. **`GET /users/abc` 500s on the golden page.** `parse-long` returns nil on garbage, nil goes into `$1`, and the beginner's showcase handler detonates. You made coercion visible — good, I even respect it — but then you have to *finish the thought* on the sacred page, because the sacred page is the curriculum. Demand: show the not-found/bad-input branch (or make the funnel map the nil-param case), because right now the first thing a beginner learns from fifteen blessed lines is a stack trace via URL typo.

13. **You specced cross-provider AI fallback chains for an interpreter that cannot serve HTTP.** The honesty note — which I do commend — admits `cljgo new && cljgo dev` boots *nothing today*, and the flagship live-handler demo ran through a Go bridge because the runtime can't even see net/http. Meanwhile T4 lovingly details step-key model resolution, native JSON modes, per-call timeouts, and an interaction-log seam across three vendor SDKs. This is cathedral-window shopping while the foundation is a rendering. Rails 0.5 was a working web framework and nothing else. Demand: nothing below T1 gets another paragraph of design until the seed registry gap is closed and a generated app actually boots. Cut T4 from this change entirely; "independently versioned satellite" is already the argument for doing exactly that.

14. **"One static binary" that can't leave home without three suitcases.** The deployment pitch is the compiled binary. The app needs `conf.edn`, `public/`, and `migrations/` at runtime. Are they embedded? Copied alongside? Is there a `cljgo build` that produces a deployable artifact, and does the binary run its own migrations on boot or is that an ops step? The production checklist is a T3 *guide*, and the actual deployment mechanics are specified nowhere. The single-binary story is cljgo's best trump card over Rails — capsules, Docker, asset compilation, the whole sad parade — and this draft plays it face-down. Demand: a Deployment requirement with scenarios (embed public/ and migrations by default; `./myapp migrate && ./myapp serve`), gating T2, not a pamphlet at T3.

15. **The spec cites its own reviews as evidence, and shipped with the placeholders showing.** The VERDICT's header claims it was "hardened by three adversarial DHH-persona review rounds"; scroll down and Round 3 is an empty section reading "(recorded after the round)" and the owner questions say "(populated after round 3)." So the document asserts, in its conclusion, the existence of scrutiny it hasn't received yet. That's the tell of a spec grading its own homework — and worse, it means the "YES" at the top was written before the process it invokes as justification finished. Also, minor but symptomatic: `:ttl "5m"` — Go duration strings smuggled into EDN as stringly-typed config in a Lisp with a reader you own. You built a reader! Demand: strike the self-citation from the verdict, fill the holes before anyone calls this reviewed, and give durations a literal or a number.

**Verdict.** Would this make me quit Rails? No — but it's the first Clojure framework spec I've read where the *taste* is mostly right, which is why the failures make me angry instead of bored. The instincts are genuinely good: generator as convention-carrier, security in the defaults, jobs in the database, no ORM theology dressed as pragmatism, guides gating releases like code — someone here has actually read the Doctrine instead of just quoting it. But the golden page — the artifact the entire draft stakes itself on — currently violates its own no-I/O rule, promises a shutdown wiring that doesn't exist, 500s on a typo'd URL, gets 403'd by its own middleware, and boots against a database nobody told the beginner to install; and the spec contradicts itself on Results, tiers, and dev/prod parity in ways a single careful read-through catches. That's not a framework, that's a mood board with SHALL statements. The difference between Rails and every "Rails of X" that died in a blog post is that Rails' golden path *ran* — every line of it, on day one, on a laptop, against a database that came in the box. Close the interpreter gap, provision the database, make the sacred page actually pass its own spec, and this becomes the most interesting web framework pitch in a decade. Ship it as written and it's Biff with a better logo.
