# S13 VERDICT — numeric-tower divergence table

276 probes total (266 in `probes.clj` + 10 in `probes_abs.clj`, split only
because `abs` fails to *resolve* in cljgo, which would otherwise abort every
form after it in one file). **89/266 diverge** in `probes.clj` (10/10 in
`probes_abs.clj`, i.e. `abs` is simply missing) — raw output in `out/`,
generated by `run_probes.sh` + `diff_probes.py`. Every row below was run
against both `clojure -M` (real 1.12.5, the oracle) and
`go run ./cmd/cljgo run` on this checkout.

## Divergence table (by cluster)

### Cluster A — `quot`/`mod`/`rem` don't guard `##Inf`/`##NaN` (14 probes)

| probe | oracle | cljgo |
|---|---|---|
| `(quot ##Inf 1)` | `THREW:Infinite or NaN` | `9223372036854775807M` |
| `(quot ##-Inf 1)` | `THREW:Infinite or NaN` | `-9223372036854775808M` |
| `(quot ##NaN 1)` | `THREW:Infinite or NaN` | `0M` |
| `(quot 1 ##NaN)` | `THREW:Infinite or NaN` | `0M` |
| `(mod ##Inf 1)` / `(mod ##-Inf 1)` / `(mod ##NaN 1)` / `(mod 1 ##NaN)` | `THREW:Infinite or NaN` | `##NaN` |
| `(mod 1 ##Inf)` | `##NaN` | `1.0` |
| `(mod 1 ##-Inf)` | `##NaN` | `##-Inf` |
| `(rem ##Inf 1)` / `(rem ##-Inf 1)` / `(rem ##NaN 1)` / `(rem 1 ##NaN)` | `THREW:Infinite or NaN` | `##NaN` |
| `(rem 1 ##Inf)` / `(rem 1 ##-Inf)` | `##NaN` | `1.0` |
| `(rem 10.0 0)` / `(mod 10.0 0)` | `THREW:Divide by zero` | `##NaN` |

**Root cause:** `float64Ops.Quotient` (`pkg/lang/numberops.go:644`) checks only
`IsZero(yf)` before dividing; when the quotient is out of int64 range (which
`##Inf`/`##NaN` always are) it falls through to
`AsBigDecimal(AsBigInt(q))` — see cluster B for why that path is itself
broken — instead of raising `ArithmeticException("Infinite or NaN")` the way
JVM Clojure's `Numbers.quotient` does. `float64Ops.Remainder`
(`pkg/lang/numberops.go:656`) is worse: it's a bare `math.Mod(x, y)` with
*no* zero/Inf/NaN check at all, so `(rem 10.0 0)` silently returns `NaN`
instead of throwing, and `(rem 1 ##Inf)` returns `1.0` (math.Mod's own
semantics) instead of Clojure's `##NaN`.

### Cluster B — `AsBigInt(float64)` saturates instead of exact conversion (2 probes, but shared machinery with A)

| probe | oracle | cljgo |
|---|---|---|
| `(bigint 1.7976931348623157e+308)` (Double/MAX_VALUE) | `179769...368N` (exact 309-digit integer) | `9223372036854775807N` (== int64 max) |
| `(quot ##Inf 1)` | (see cluster A) | saturates the same way |

**Root cause:** `AsBigInt` (`pkg/lang/numberops.go:821`, `float64`/`float32`
cases at ~843) does `NewBigIntFromInt64(int64(x))` — a naive Go float→int64
conversion, which for any float outside int64 range clamps/saturates rather
than producing the mathematically exact big integer (Java's
`BigInteger` conversion from a `double` is always exact). This is the single
function behind both the `bigint`-from-huge-double bug and the
`quot`/Inf-argument garbage-value bug in cluster A.

### Cluster C — `BigDecimal` has no scale; can represent Inf/NaN (11 probes)

| probe | oracle | cljgo |
|---|---|---|
| `(quot 10.0M 3)` | `3.0M` | `3M` |
| `(mod 10.0M 3)` / `(mod 10.0M 3.0M)` | `1.0M` | `1M` |
| `(rem 10.0M 3)` / `(rem 10.0M 3.0M)` | `1.0M` | `1M` |
| `(bigdec 1.0)` / `(bigdec 1.0M)` | `1.0M` | `1M` |
| `(bigdec "1e10")` | `1E+10M` | `10000000000M` |
| `(bigdec "1e-10")` | `1E-10M` | `0.0000000001M` |
| `(bigdec -0.0)` | `0.0M` (sign normalized) | `-0M` |
| `(bigdec ##Inf)` / `(bigdec ##-Inf)` | `THREW:Infinite or NaN` | `+InfM` / `-InfM` |
| `(bigdec ##NaN)` | `THREW:Infinite or NaN` | `THREW:Float.SetFloat64(NaN)` (internal Go method name leaks into the message) |

**Root cause:** `pkg/lang/bigdecimal.go:19` — `BigDecimal` wraps a Go
`big.Float`, and the file already carries the maintainers' own TODO:
*"swap out with a decimal representation... big.Float is a binary floating
point representation... can lead to unexpected results."* `big.Float` has
no scale/precision-preserving decimal semantics (so trailing zeros and
exponential notation are lost) and, unlike `java.math.BigDecimal`, it *can*
represent `+Inf`/`-Inf`/`NaN` — which is why `bigdec`/`quot`/`mod`/`rem`
silently produce those instead of throwing. This is one representation
choice driving all 11 rows plus 3 of the suite's named `bigdec` fails.

### Cluster D — `even?`/`odd?` have no integer-type guard (12 probes)

| probe | oracle | cljgo |
|---|---|---|
| `(even? ##Inf)` / `##-Inf` / `##NaN` / `1.5` / `0.2M` / `1/2` | `THREW:Argument must be an integer: <x>` | `false` (computed anyway) |
| `(odd? ##Inf)` / `##NaN` / `1.5` / `1/2` / `0.2M` | `THREW:Argument must be an integer: <x>` | `true` |
| `(even? nil)` / `(odd? nil)` | `THREW:Argument must be an integer: ` | `THREW:cannot convert <nil> to Ops` (wrong message, but still throws) |

**Root cause:** `core/core.clj:882-883`:

```clojure
(defn even? [n] (zero? (rem n 2)))
(defn odd? [n] (not (even? n)))
```

Real Clojure's `even?`/`odd?` explicitly check `(integer? n)` first and
throw `ArithmeticException "Argument must be an integer: ..."` otherwise.
cljgo's version just calls `rem`, which happily computes on doubles, ratios,
bigdecimals and `##Inf`/`##NaN` (per Cluster A, `rem` doesn't even validate
its own arguments) and returns a bogus boolean instead of throwing. This
single 2-line definition accounts for the entire even?/odd? suite failure
bucket (6 + 6 = 12).

### Cluster E — `abs` is not registered as a builtin at all (10 probes)

`(abs -1)` fails to *resolve* (`unable to resolve symbol: abs`), not merely
to compute the wrong thing — a compile-time error, so it aborts every form
after it in a shared file (why `probes_abs.clj` is separate).

**Root cause:** every numeric `Ops` implementation in
`pkg/lang/numberops.go` already has a working `Abs` method (e.g.
`float64Ops.Abs` at `numberops.go:684`, `math.Abs(AsFloat64(x))`) — the
implementation exists per-type. `abs` is simply never `def`'d as a
top-level builtin (no entry in `pkg/eval/numeric_builtins.go` or
`predicate_builtins.go`, no `core.clj` wrapper). This is pure wiring, not a
semantics gap — the cheapest fix in the whole table.

### Cluster F — `int` bounds-checks against 64-bit range, not 32-bit (4 probes)

| probe | oracle | cljgo |
|---|---|---|
| `(int -2147483648.000001)` | `THREW:Value out of range for int` | `-2147483648` (silently truncated) |
| `(int -2147483649)` | `THREW:integer overflow` | `-2147483649` |
| `(int 2147483648)` | `THREW:integer overflow` | `2147483648` |
| `(int 2147483647.000001)` | `THREW:Value out of range for int` | `2147483647` |

**Root cause:** `IntCast` (`pkg/lang/numbers.go:949`) bounds-checks integral
inputs against `math.MinInt`/`math.MaxInt` — Go's *platform* `int`, which on
every build target cljgo ships for is 64-bit — instead of
`math.MinInt32`/`math.MaxInt32`. Java's `int` is always 32 bits regardless
of platform, so JVM Clojure's `int` throws well inside the 64-bit range;
cljgo's never does. (The `float64`/`float32` branches of the same function
do use the right conceptual bound but reuse the same `math.MinInt`/
`math.MaxInt` constants, so they inherit the bug too.)

### Cluster G — exception messages differ in wording only (cosmetic, ~35 probes)

`"long overflow"` vs `"integer overflow"`, `"/ by zero"` vs `"runtime error:
integer divide by zero"`, `"Divide by zero"` vs `"division by zero"` /
`"divide by zero"`, `"Cannot invoke \"Object.getClass()\"..."` vs `"cannot
convert <nil> to Ops"`, and similar. In every one of these an exception *is*
thrown by both implementations, and the suite's own assertions
(`p/thrown?`) only check that something was thrown, not its message text —
so this cluster, while real, is not implicated in any of the suite's
numeric fails. Lowest priority; cosmetic polish only.

## Recommended scope for ADR 0029

**In scope (drives the suite's numeric fail counts directly):**

1. Cluster D (`even?`/`odd?` integer guard) — 2-line fix in `core.clj`,
   closes 12 fails outright.
2. Cluster A (`quot`/`mod`/`rem` Inf/NaN guards in `float64Ops`) — closes
   most of the 30+25+23 quot/mod/rem bucket; requires explicit
   `math.IsInf`/`math.IsNaN` checks before the int64-range fast path in
   `Quotient`, and a real zero/Inf/NaN check in `Remainder` (currently a
   bare `math.Mod`).
3. Cluster F (`int` 32-bit bounds) — swap `math.MinInt`/`math.MaxInt` for
   `math.MinInt32`/`math.MaxInt32` in the relevant `IntCast` branches.
4. Cluster E (`abs` wiring) — register the existing per-type `Ops.Abs` as a
   top-level builtin; trivial, high suite-coverage-per-effort ratio.
5. Cluster B (`AsBigInt(float64)` exact conversion) — needed for cluster A
   to fully close (quot's `##Inf`/`##-Inf` args still route through this
   function even after adding a guard, to avoid *other* large-double
   `bigint`/`quot` paths silently saturating) and for the `bigint` suite
   fail. Fix: convert via `big.Float.SetFloat64(x).Int(nil)` (exact) instead
   of `int64(x)` when `x` is outside int64 range.

**Larger, deferred (real, but a bigger lift than a "spike closes it" size):**

6. Cluster C (`BigDecimal` representation) — the maintainers' own TODO
   already flags this. Fixing it properly means swapping `big.Float` for a
   real scaled-decimal representation (unscaled `*big.Int` + int scale, à la
   `java.math.BigDecimal`, or an existing Go decimal library) — a
   redesign of `pkg/lang/bigdecimal.go`, not a guard-clause fix. Recommend
   a **separate ADR** (not folded into 0029) once 0029's cheaper fixes are
   shipped, since it touches `NewBigDecimal`, every `bigDecimalOps` method,
   and the `pr-str`/emitter constant-reconstruction path
   (`MustBigDecimal`) together.

**Out of scope:**

7. Cluster G (message-wording cosmetics) — real but doesn't move the suite
   number; can be picked up opportunistically, not worth its own ADR.
